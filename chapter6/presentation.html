Designing Data-Intensive Applications

Partitioning

Knowledge Transfer Session • Chapter 6

"We must break away from the sequential and not limit the computers."

— Grace Murray Hopper, Management and the Computer of the Future (1962)

For very large datasets or very high throughput, replication alone isn't enough. We need to break data into partitions (a.k.a. shards, regions, tablets, vnodes).

Why Partition?

Scalability
Spread data and query load across many nodes

The Goal

10 nodes should handle 10× the data and 10× the throughput of a single node.

The Enemy: Hot Spots 🔥

Skewed partitioning → disproportionate load on one partition → 9 idle nodes, 1 bottleneck.

Partitioning Strategies

Key range vs hash — and their trade-offs

Partitioning + Replication

Usually combined: each partition is replicated on multiple nodes for fault tolerance.

  • Each node may store multiple partitions
  • A node can be leader for some partitions, follower for others
  • Partitioning scheme is independent of replication scheme

Partitioning by Key Range

Assign continuous ranges of keys to each partition — like volumes of an encyclopedia.

  • Ranges adapt to data distribution (not evenly spaced)
  • Within each partition, keys are sorted → range scans are efficient
  • Great for time-series data: fetch all readings from a month

Used by: HBase Bigtable RethinkDB

Key Range: The Hot Spot Problem

If key is a timestamp, all writes go to the partition for today → hot spot!

❌ Problem

Key: 2024-01-15-14:30:05 ← all writes hit today's partition

✅ Solution

Key: sensor_42/2024-01-15-14:30:05 ← partition first by sensor, then time

Spread writes across sensors, but now range queries need per-sensor requests.

Partitioning by Hash of Key

Apply a hash function to each key → assign ranges of hashes to partitions.

  • Even similar keys → different hashes → evenly distributed
  • Hash functions: MD5 (MongoDB) Murmur3 (Cassandra) FNV (Voldemort)
  • ⚠️ Language built-in hashes (Java hashCode) may differ across processes!
Trade-off
Lose the ability to do efficient range queries

Cassandra's Compound Key Compromise

Best of both worlds: hash the first part, sort by the rest.

PRIMARY KEY (user_id, update_timestamp) ↑ hashed for partition ↑ sorted within partition
  • Can't range-scan across user_id (it's hashed)
  • Can efficiently range-scan timestamps within a single user
  • Elegant model for one-to-many relationships (user → posts, user → events)

Skewed Workloads: The Celebrity Problem

Hashing distributes keys but can't help when all requests target the same key.

A celebrity with millions of followers posts → massive writes to one key (their user ID or post ID).

Workaround: add a random number to the key:

key = "celebrity_post_123" + random(0, 99) → splits writes across 100 keys on different partitions

⚠️ Now reads must query all 100 keys and combine results. Only worth it for the small number of truly hot keys.

Today: most systems require application-level handling. Future: automatic detection.

Partitioning & Secondary Indexes

The bread and butter of relational databases meets partitioning

Document-Partitioned Indexes

Each partition maintains its own local index.

  • ✅ Writes: only update one partition
  • ❌ Reads: must scatter/gather across all partitions
  • Prone to tail latency amplification

Example: searching for red cars → query every partition, combine results.

Used by: MongoDB Cassandra Elasticsearch Riak

Term-Partitioned Indexes

A global index covering all partitions, itself partitioned by term.

  • ✅ Reads: query one partition of the index
  • ❌ Writes: may affect multiple partitions (distributed transaction)
  • In practice: updates are often async

Example: color:red entries from ALL data partitions live in one index partition.

Used by: DynamoDB Riak Search Oracle DW

Local vs Global Indexes

Document-Partitioned (Local) Term-Partitioned (Global)
Writes✅ One partition❌ Multiple partitions
Reads❌ Scatter/gather✅ One partition
ConsistencyImmediateOften async
ComplexitySimplerDistributed txn needed

Choose based on your read/write ratio and consistency requirements.

Rebalancing Partitions

Moving data when nodes join or leave the cluster

Why Rebalance?

  • 📈 Query throughput increases → add more CPUs
  • 💾 Dataset grows → add more disks and RAM
  • 💥 Machine fails → other nodes take over

Requirements for rebalancing:

⚖️ Fair

Load evenly shared after rebalancing

🔄 Available

DB continues serving reads/writes during rebalancing

📦 Minimal

Move only what's necessary (minimize network I/O)

❌ Don't: hash mod N

hash(key) mod N → partition number

Seems simple, but when N changes, most keys move:

hash(key) = 123456 mod 10 = 6 → node 6 mod 11 = 3 → node 3 ← moved! mod 12 = 0 → node 0 ← moved again!

Adding one node causes a massive data shuffle. Excessively expensive. Need better approaches.

Fixed Number of Partitions

Create many more partitions than nodes upfront.

  • 10 nodes, 1,000 partitions → ~100 per node
  • New node steals some partitions from existing nodes
  • Only entire partitions move — keys stay in their partition

Used by: Riak Elasticsearch Couchbase Voldemort

⚠️ Must choose partition count wisely — it's fixed forever. Too many = overhead, too few = limits growth.

Dynamic & Proportional Partitioning

🔀 Dynamic Partitioning

Partition grows too large → split in half. Shrinks → merge with neighbor.

  • Adapts to data volume automatically
  • ⚠️ Empty DB starts with 1 partition (need pre-splitting)
  • HBase default: split at 10 GB

Used by: HBase MongoDB RethinkDB

📏 Proportional to Nodes

Fixed number of partitions per node.

  • More nodes → partitions split → each gets smaller
  • New node randomly splits existing partitions
  • Cassandra: 256 partitions per node

Used by: Cassandra Ketama

Automatic vs Manual Rebalancing

🤖 Automatic

System decides when and where to move partitions.

✅ Less operational work

❌ Can be unpredictable

❌ Dangerous with auto failure detection: overloaded node declared dead → rebalancing adds more load → cascading failure! 💀

👤 Manual (with suggestions)

System proposes a plan, admin approves.

✅ Human in the loop prevents surprises

❌ Slower

Used by: Couchbase Riak Voldemort

Request Routing

How clients find the right node: service discovery for databases

Three Routing Approaches

  1. Contact any node — node handles or forwards
  2. Routing tier — partition-aware load balancer
  3. Smart client — client knows partition→node mapping

Key challenge: how does the routing component learn about partition reassignments?

ZooKeeper for Coordination

Separate coordination service maintains the authoritative partition→node mapping.

  • Nodes register in ZooKeeper
  • Routing tier / clients subscribe to changes
  • Partition changes → ZooKeeper notifies subscribers

Used by: Espresso (LinkedIn) HBase Kafka SolrCloud

Alternative: Gossip protocol (Cassandra, Riak) — no external dependency, but more complexity in DB nodes.

Summary

Partitioning approaches, indexes, rebalancing, and routing

Chapter 6 Summary

Partitioning Strategies

  • Key range — sorted, range queries, but hot spot risk
  • Hash — even distribution, but no range queries
  • Compound key — hash + sort (Cassandra)

Secondary Indexes

  • Local (document) — fast writes, scatter/gather reads
  • Global (term) — fast reads, complex writes

Rebalancing

  • Fixed partitions, dynamic splitting, or per-node
  • Never use hash mod N!
  • Human-in-the-loop is safer

Request Routing

  • Any-node, routing tier, or smart client
  • ZooKeeper or gossip protocol
