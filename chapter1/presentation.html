Who Is This Book For?

If you work with data at scale, this book is for you:

  • Backend Engineers — Building services that store and process data
  • Data Engineers — Designing pipelines and data infrastructure
  • System Architects — Making technology choices that scale
  • DevOps/SREs — Operating and troubleshooting data systems
  • Tech Leads — Evaluating tradeoffs in system design

Why care? Understanding how data systems work helps you make better decisions, debug faster, and build more reliable systems.

Data-Intensive vs Compute-Intensive

Today, the bottleneck is rarely CPU power.

The bigger problems are usually:

  • The amount of data
  • The complexity of data
  • The speed at which it changes

The Standard Building Blocks

Most applications need these data systems:

  • Databases — Store and retrieve data
  • Caches — Speed up expensive reads
  • Search Indexes — Filter and find data
  • Stream Processing — Handle async messages
  • Batch Processing — Crunch large datasets

Like LEGO bricks — simple pieces that combine into complex systems.

Blurring Boundaries

Traditional categories no longer fit neatly:

  • Redis — A datastore used as a message queue
  • Apache Kafka — A message queue with database-like durability

Why lump them all under "Data Systems"? Because modern applications combine many specialized tools.

Composite Data Systems

A single tool can no longer meet all needs. Applications now stitch together:

  • Primary database
  • Cache layer (Memcached)
  • Full-text search (Elasticsearch)

You become not just a developer, but a data system designer.

The Hard Questions

When designing composite data systems:

  • How do you ensure data remains correct and complete when things go wrong?
  • How do you provide consistently good performance even when parts degrade?
  • How do you scale to handle increased load?
  • What does a good API look like?

The Three Concerns

Reliability • Scalability • Maintainability

Three Concerns for Software Systems

1. Reliability

Continue working correctly even when things go wrong.

2. Scalability

Reasonable ways to deal with growth.

3. Maintainability

Many people can work on it productively over time.

Reliability

Making systems work correctly, even when faults occur

What is Reliability?

Typical expectations for reliable software:

  • Performs the function users expected
  • Tolerates user mistakes and unexpected usage
  • Performance is good enough under expected load
  • Prevents unauthorized access and abuse

Reliability = "continuing to work correctly, even when things go wrong."

Faults vs Failures

Fault: One component deviating from its spec.

Failure: The system as a whole stops providing service to users.

Think of a flat tire: A fault (puncture) doesn't have to cause failure (car stops) if you have fault tolerance (spare tire, roadside assistance).

Goal: Build fault-tolerant systems that prevent faults from causing failures.

Chaos Engineering: Netflix Chaos Monkey

"By deliberately inducing faults, you ensure that the fault-tolerance machinery is continually exercised and tested."

Many critical bugs come from poor error handling. Chaos Monkey randomly kills processes in production to:

  • Verify fault-tolerance actually works
  • Build confidence that faults will be handled correctly
  • Find weaknesses before real failures do

1. Hardware Faults

Hard disks crash. RAM fails. Power goes out. Networks disconnect.

10-50 years
Mean Time to Failure (MTTF) for hard disks

With 10,000 disks → expect ~1 disk to die per day.

Traditional Approach: Hardware Redundancy

Add redundancy to individual components:

  • RAID — Disk redundancy (Redundant Array of Independent Disks)
  • Dual power supplies — Power redundancy
  • Hot-swappable CPUs — Compute redundancy
  • Batteries & generators — Backup power

This kept machines running for years without interruption.

The Shift: Software Fault Tolerance

Cloud computing changed the game:

  • Cloud VMs can become unavailable without warning
  • Platforms prioritize flexibility over single-machine reliability
  • More machines = more frequent hardware faults

Modern approach: Design systems to tolerate entire machine loss via software coordination.

Operational Advantage: Rolling Upgrades

Systems tolerant of machine failure can be patched one node at a time, without downtime.

  • No planned downtime for security patches
  • Gradual rollout of changes
  • Ability to roll back quickly if issues arise

2. Software Errors

Systematic bugs are harder to anticipate than hardware faults.

Unlike hardware (random & independent), software bugs are correlated across nodes → more system failures.

We've all seen the dreaded Blue Screen of Death...

Examples of Software Errors

  • Leap second bug (June 30, 2012) — Linux kernel bug caused many applications to hang simultaneously
  • Runaway processes — Using up CPU, memory, disk, or network
  • Cascading failures — One fault triggers another, which triggers more
  • Slow dependencies — A service slows down or returns corrupted responses

These bugs often lie dormant until unusual circumstances trigger them.

Dealing with Software Errors

No quick solution, but many small things help:

  • Carefully think about assumptions and interactions
  • Thorough testing
  • Process isolation
  • Allow processes to crash and restart
  • Measure, monitor, and analyze behavior in production
  • Self-checking systems that raise alerts on discrepancies

3. Human Errors

Humans are unreliable. Even with best intentions, mistakes happen.

Leading Cause
Configuration errors cause most outages (not hardware!)

Hardware faults account for only 10-25% of outages.

Mitigating Human Errors

  • Design for safety — Well-designed APIs that make "the right thing" easy
  • Sandbox environments — Let people experiment without affecting production
  • Thorough testing — Unit, integration, and manual tests; cover corner cases
  • Quick recovery — Fast rollbacks, gradual rollouts (canary releases)
  • Monitoring & telemetry — Early warning signals, track assumptions
  • Good training — Management practices and procedures

How Important is Reliability?

Not just for nuclear power stations!

  • Business apps: Lost productivity, legal risks
  • E-commerce: Lost revenue, reputation damage
  • Personal data: Would users know how to restore from backup?

"Consider a parent who stores all their pictures and videos of their children in your photo application..."

You may sacrifice reliability for cost sometimes, but be conscious when cutting corners.

Scalability

Strategies for keeping performance good as load increases

What is Scalability?

It's not a one-dimensional label.

❌ "X is scalable" or "Y doesn't scale"

✅ "If the system grows in X way, what are our options for coping with that growth?"

Scalability is about asking the right questions, not checking a box.

Describing Load

First, describe the current load with Load Parameters:

  • Requests per second (RPS)
  • Database read/write ratio
  • Simultaneously active users
  • Cache hit rate

The best parameters depend on your system's architecture. Let's look at a real example...

Case Study: Twitter (2012)

Two main operations:

4.6k/sec
Post Tweet (avg), 12k peak
300k/sec
Home Timeline reads

12k writes/sec is manageable. So what's the real challenge?

The Challenge: Fan-out

The problem isn't tweet volume — it's fan-out.

Fan-out: The number of requests to other services needed to serve one incoming request.

Each user follows many people. Each user is followed by many people. When someone tweets, how do you get it to all their followers' timelines?

Twitter tried two approaches...

Approach 1: Query on Read

Insert tweets into a global collection. On read, query and merge:

SELECT tweets.*, users.*
FROM tweets
JOIN users ON tweets.sender_id = users.id
JOIN follows ON follows.followee_id = users.id
WHERE follows.follower_id = current_user

✅ Simple writes

❌ Expensive reads — Every timeline load hits the database hard

Approach 1: The Problem

With 300,000 timeline reads/second, the database couldn't keep up.

Each timeline read requires:

  • Finding all users you follow
  • Finding all their tweets
  • Merging and sorting by time

Twitter needed a new approach...

Approach 2: Pre-compute Timelines

Maintain a cache for each user's home timeline (like a mailbox).

When a user posts a tweet, write it to all their followers' caches.

✅ Cheap reads — Timeline already computed!

❌ Expensive writes — One tweet = many writes

This worked better! Reads are ~2 orders of magnitude more frequent than writes.

The "Trump Problem" 🐦

Approach 2 has a catch...

Average tweet → 75 followers → 75 writes

4.6k tweets/sec × 75 = 345k writes/sec

But some users have 30+ million followers!

One celebrity tweet = 30 million writes to home timeline caches.

Twitter tries to deliver within 5 seconds. 😰

The Hybrid Solution ✨

Twitter now uses both approaches:

🚀Explore the Twitter PoC Code

👥
Regular users: Approach 2
(fan-out on write)
⭐
Celebrities: Approach 1
(merged on read)

The distribution of followers per user is a key load parameter for Twitter's scalability.

Describing Performance

Once you've described load, investigate what happens when it increases:

  • Option 1: Keep resources constant → how is performance affected?
  • Option 2: Keep performance constant → how many resources needed?

Metrics differ by system type:

  • Batch processing: Throughput (records/second)
  • Online systems: Response time

Latency vs Response Time

Often used interchangeably, but they're different:

Response Time

What the client sees: service time + network delays + queueing delays

Latency

Time a request is waiting to be handled (awaiting service)

Response Time Distribution

Response time varies. Think of it as a distribution, not a single number.

Causes of variation:

  • Context switches
  • Network packet loss & TCP retransmission
  • Garbage collection pauses
  • Page faults reading from disk

Why Averages Can Be Misleading

Imagine Jeff Bezos walks into a room with 1,000 people, each earning $50,000/year.

$1,048,951
Average income in the room

(1,000 × $50,000 + $1,000,000,000) ÷ 1,001 = $1,048,951

The average tells you nothing about the typical person's experience. Same problem with response times!

Percentiles: p50, p95, p99, p99.9

Median (p50): Half of requests faster, half slower. The "typical" experience.

High percentiles (tail latencies):

  • p95: 95% of requests are faster than this
  • p99: 99% of requests are faster than this
  • p99.9: 99.9% of requests are faster than this

Average (mean) is not a good metric — it doesn't tell you how many users experienced that delay.

Why p99.9 Matters: The Amazon Effect

Amazon measures requirements at the 99.9th percentile.

Why? Customers with slowest requests often have the most data = most valuable customers.

1%
Sales drop per 100ms slower
16%
Satisfaction drop per 1s slower
p99.99?
Too expensive, diminishing returns

Optimizing the 99.99th percentile was deemed not worth the effort for Amazon.

Tail Latency Amplification

One end-user request may call multiple backend services in parallel.

The end-user must wait for the slowest call.

Even if only a small % of backend calls are slow, more calls = higher chance of hitting a slow one.

Head-of-Line Blocking

A server can only process a limited number of things in parallel.

A few slow requests can hold up subsequent requests in the queue.

Even if subsequent requests are fast to process, clients see slow response times due to queueing delays.

Important: Measure response times on the client side.

SLOs and SLAs

Service Level Objectives (SLOs) and Service Level Agreements (SLAs) define expected performance.

Example SLA:

  • Median response time < 200ms
  • 99th percentile < 1 second
  • Service up 99.9% of the time

If SLA is not met, customers may demand a refund.

Percentiles in Practice

Calculate percentiles efficiently with algorithms like:

  • Forward decay
  • t-digest
  • HdrHistogram

Warning: Averaging percentiles is mathematically meaningless!

The right way: aggregate the histograms, then calculate percentiles.

Approaches for Coping with Load

Scaling Up (Vertical)

Move to a more powerful machine.

✅ Simple

❌ Expensive, hits ceiling

Scaling Out (Horizontal)

Distribute load across multiple machines.

✅ No ceiling

❌ More complex

In reality, good architectures use a pragmatic mix of both.

Elastic vs Manual Scaling

Elastic systems: Automatically add resources when load increases.

  • Useful for highly unpredictable load

Manual scaling: Humans analyze capacity and decide to add machines.

  • Simpler, fewer operational surprises

Stateful vs Stateless Services

Stateless services: Easy to distribute across machines.

Stateful data systems: Much more complex to distribute.

Common wisdom: Keep your database on a single node until scaling cost or high-availability forces you to distribute.

This is changing as distributed system tools improve.

No Magic Scaling Sauce

Large-scale architectures are highly specific to the application.

The problem may be:

  • Volume of reads or writes
  • Volume of data to store
  • Complexity of data
  • Response time requirements
  • Access patterns

A system for 100k req/sec (1KB each) looks very different from 3 req/min (2GB each).

Maintainability

Making life better for engineering and operations teams

The Cost of Maintenance

The majority of software cost is not initial development.

Ongoing maintenance includes:

  • Fixing bugs
  • Keeping systems operational
  • Investigating failures
  • Adapting to new platforms
  • Repaying technical debt
  • Adding new features

Three Design Principles

1. Operability

Make it easy for ops to keep the system running.

2. Simplicity

Make it easy for new engineers to understand.

3. Evolvability

Make it easy to adapt for future requirements.

Operability: Life Easy for Ops

"Good operations can work around bad software, but good software cannot run reliably with bad operations."

Good operability means:

  • Good monitoring & visibility into runtime behavior
  • Support for automation and standard tools
  • Avoiding dependency on individual machines
  • Good documentation and operational model
  • Good defaults with ability to override
  • Self-healing with manual control when needed
  • Predictable behavior, minimizing surprises

Simplicity: Managing Complexity

Symptoms of complexity:

  • Explosion of state space
  • Tight coupling of modules
  • Tangled dependencies
  • Inconsistent naming
  • Performance hacks everywhere

Accidental complexity: Not inherent in the problem, arises from implementation.

Best tool: Good abstractions that hide implementation details.

Evolvability: Making Change Easy

Requirements will change. Count on it.

  • New facts are learned
  • Unanticipated use cases emerge
  • Business priorities shift
  • Platforms change
  • Legal/regulatory requirements change

Closely linked to simplicity: simple systems are easier to modify.

Also called: extensibility, modifiability, plasticity.

Summary

Chapter 1 Summary

Reliability: Make systems work correctly even when faults occur (hardware, software, human). Use fault-tolerance techniques.

Scalability: Describe load (load parameters), measure performance (percentiles), and build strategies to handle growth.

Maintainability: Focus on Operability, Simplicity, and Evolvability. Good abstractions reduce complexity.

No easy fixes — but patterns and techniques reappear across different applications. That's what the rest of the book covers.
