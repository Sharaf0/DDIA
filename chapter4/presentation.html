Formats for Encoding Data

From in-memory objects to bytes on the wire

Two Representations of Data

🧠 In Memory

Objects, structs, lists, hash tables, trees...

Optimized for CPU access via pointers

📡 On the Wire / Disk

Self-contained sequence of bytes

Pointers are meaningless to other processes

Translation: Encoding (serialization/marshalling) ↔ Decoding (parsing/deserialization/unmarshalling)

Language-Specific Formats ❌

Java's Serializable, Ruby's Marshal, Python's pickle, Kryo...

  • Tied to one language — can't read the data from another language
  • Security risk — decoding can instantiate arbitrary classes → remote code execution 💀
  • Versioning is an afterthought — poor forward/backward compatibility
  • Poor efficiency — Java serialization is notorious for bloated encoding

"It's generally a bad idea to use your language's built-in encoding for anything other than very transient purposes."

JSON, XML, and CSV

Widely known, widely supported, and almost as widely disliked. 😅

  • Number ambiguity — JSON can't distinguish integers from floats; integers > 2⁵³ break in JavaScript (Twitter includes tweet IDs twice!)
  • No binary strings — must Base64 encode → 33% size increase
  • Schema support — optional for XML/JSON, nonexistent for CSV
  • CSV quirks — commas in values? Newlines? Escaping rules aren't always followed

Despite flaws, they're good enough for data interchange between organizations. Getting people to agree on anything outweighs most concerns.

Binary Encoding: MessagePack

JSON/XML are verbose → binary variants exist: MessagePack BSON BJSON Smile

Example record:

{ "userName": "Martin", "favoriteNumber": 1337, "interests": ["daydreaming", "hacking"] }

MessagePack: 66 bytes vs JSON text: 81 bytes

Only ~18% saving. Still includes field names. We can do much better...

Thrift & Protocol Buffers

Schema-driven binary encoding from Facebook & Google

Schema Definitions

Thrift IDL

struct Person { 1: required string userName, 2: optional i64 favoriteNumber, 3: optional list<string> interests }

Protocol Buffers

message Person { required string user_name = 1; optional int64 favorite_number = 2; repeated string interests = 3; }

Both use numbered field tags instead of field names → much more compact encoding

Thrift BinaryProtocol

Each field encoded with:

  • Type annotation (string, integer, list...)
  • Field tag number (1, 2, 3) instead of names
  • Length indication where needed
59 bytes
vs 66 for MessagePack, 81 for JSON text

Thrift CompactProtocol

Packs field type + tag into a single byte

Uses variable-length integers: numbers -64 to 63 fit in 1 byte, -8192 to 8191 in 2 bytes

34 bytes
Same data, ~half the size of BinaryProtocol

Protocol Buffers Encoding

Similar bit packing to Thrift CompactProtocol

33 bytes
Most compact of Thrift/Protobuf encodings

required vs optional makes no difference in encoding — it's only a runtime validation check

Field Tags & Schema Evolution

Field tags are critical — they're how encoded data identifies fields.

✅ You CAN

  • Change field names (tags stay the same)
  • Add new fields with new tag numbers
  • Remove optional fields

❌ You CANNOT

  • Change a field's tag number
  • Make a new field required
  • Remove a required field
  • Reuse a deleted field's tag number

How Compatibility Works

→ Forward Compatibility

Old code reads new data

Old code sees unknown tag numbers → skips them (datatype annotation tells how many bytes to skip)

← Backward Compatibility

New code reads old data

New code knows tag meanings. New fields must have default values since old data won't have them.

Protobuf special: optional → repeated is allowed. Old code sees only the last element.

Apache Avro

Born from Hadoop, designed for dynamically generated schemas

Avro Schema — No Tag Numbers!

Avro IDL

record Person { string userName; union { null, long } favoriteNumber = null; array<string> interests; }

JSON Representation

{ "type": "record", "name": "Person", "fields": [ {"name": "userName", "type": "string"}, {"name": "favoriteNumber", "type": ["null","long"], "default": null}, {"name": "interests", "type": {"type":"array","items":"string"}} ] }

Avro Encoding

32 bytes
Most compact encoding we've seen!

No field identifiers or type annotations in the data. Just values concatenated together.

Decoded by walking through fields in schema order. Reader MUST use the exact same schema... or must they? 🤔

Writer's Schema vs Reader's Schema

The key idea: they don't have to be the same — just compatible!

✍️ Writer's Schema

The schema used when the data was encoded. Compiled into the writing application.

📖 Reader's Schema

The schema the reading application expects. May be a different version!

Avro resolves differences by matching fields by name, filling in defaults for missing fields, and ignoring unknown ones.

Schema Resolution in Action

  • Fields matched by name (not position)
  • Different field order? No problem
  • Writer has field reader doesn't? Ignored
  • Reader expects field writer doesn't have? Filled with default

To maintain compatibility: only add/remove fields that have default values

Avro uses union types for nullability — no optional/required markers

How Does the Reader Find the Writer's Schema?

📁 Large Files

Hadoop-style: millions of records, same schema. Include schema once at file beginning.

→ Avro object container files

🗄️ Database Records

Different records may use different schemas. Include a version number per record.

→ Look up schema from version DB (e.g., Espresso)

🌐 Network

Two processes negotiate schema version on connection setup.

→ Avro RPC protocol

Avro's Superpower: Dynamically Generated Schemas

Want to dump a relational database to a binary file?

  • Generate an Avro schema from the database schema automatically
  • Each table → a record, each column → a field
  • Column name = field name (no manual tag assignment needed!)
  • Database schema changes? Just regenerate the Avro schema. No manual intervention.

With Thrift/Protobuf, you'd need to manually maintain tag-to-column mappings — "this kind of dynamically generated schema simply wasn't a design goal."

The Merits of Schemas

Schema-driven binary encodings win on multiple fronts:

  • 📦 More compact — omit field names from encoded data
  • 📝 Living documentation — schema is required for decoding, so it's always up to date
  • ✅ Compatibility checking — database of schemas lets you verify compatibility before deployment
  • 🔧 Code generation — for statically typed languages: type checking at compile time

"Schema evolution allows the same kind of flexibility as schemaless/schema-on-read JSON databases provide, while also providing better guarantees about your data and better tooling."

Modes of Dataflow

Where and how encoded data travels between processes

Three Ways Data Flows

🗄️ Through Databases

Writer encodes, reader decodes. "Sending a message to your future self."

🌐 Through Services

REST, RPC — client encodes request, server decodes and responds.

📨 Through Messages

Async message passing via brokers — sender encodes, consumer decodes.

Dataflow Through Databases

Backward compatibility is clearly needed — your future self must decode what you wrote.

Forward compatibility is also needed — during rolling upgrades, older code may read data written by newer code.

⚠️ Danger: Old code reads a record with a new field, updates it, and writes it back — the new field might be lost!

Encoding formats handle this, but watch out at the application level.

Data Outlives Code

"Within a single database you may have some values that were written five milliseconds ago, and some values that were written five years ago."

You deploy new code in minutes. But old data stays forever in its original encoding.

  • Rewriting (migrating) all data is expensive for large datasets
  • Most relational DBs: ALTER TABLE ADD COLUMN with NULL default — no rewrite needed
  • LinkedIn's Espresso uses Avro for storage → Avro schema evolution handles it

Schema evolution lets the DB appear as if all data uses one schema, even if it contains records from many eras.

Archival Storage

Database snapshots for backup or data warehousing:

  • Dump is encoded with the latest schema — a consistent snapshot
  • Written once, immutable — perfect for Avro object container files
  • Good opportunity to encode in an analytics-friendly format like Parquet (column-oriented)

Dataflow Through Services

REST, SOAP, RPC — and the pitfalls of location transparency

Services and Microservices

Servers expose an API (service); clients connect and make requests.

  • A server can be a client to another service — this is the foundation of microservices
  • Key goal: services are independently deployable and evolvable
  • Services provide encapsulation — restrict what clients can do via the API

Old and new versions of servers and clients run simultaneously → data encoding must be compatible across versions.

REST vs SOAP

REST ✅

  • Design philosophy built on HTTP principles
  • Simple data formats (JSON), URLs for resources
  • Uses HTTP features: caching, auth, content negotiation
  • API described with OpenAPI/Swagger
  • Predominant for public APIs

SOAP ⚠️

  • XML-based protocol, avoids HTTP features
  • Complex standards: WS-* framework
  • WSDL for API description → code generation
  • Vendor interoperability problems
  • Still used in large enterprises

The Problems with RPC

RPC tries to make a network call look like a local function call (location transparency). But it's fundamentally flawed:

  • 🎲 Unpredictable — network requests can be lost, slow, or timeout
  • ⏱️ Timeouts — no response ≠ failure. You just don't know what happened
  • 🔄 Retries need idempotence — the request may have succeeded; only the response was lost
  • 🐌 Wildly variable latency — sub-millisecond to multi-second for the same call
  • 📦 No pointer passing — all parameters must be serialized to bytes
  • 🌐 Cross-language type mismatches — e.g., JS can't handle integers > 2⁵³

Modern RPC Frameworks

RPC isn't going away — but new frameworks are more honest about network realities:

  • gRPC Protocol Buffers, supports streams (request/response over time)
  • Thrift Finagle Use futures/promises for async + parallel requests
  • Rest.li JSON over HTTP with futures
  • Avro RPC Schema negotiation on connection setup

REST wins for public APIs (easy debugging, broad ecosystem). RPC wins for internal services (performance, efficiency).

Evolution for RPC

Simplifying assumption: servers updated first, clients second.

  • Need backward compatibility on requests (new server reads old client's data)
  • Need forward compatibility on responses (old client reads new server's data)

Compatibility rules depend on the encoding format used (Thrift, Protobuf, JSON, etc.)

Cross-organization APIs often require indefinite compatibility — the provider can't force clients to upgrade. Common to maintain multiple API versions side by side.

Message-Passing Dataflow

Async communication via message brokers and actors

Message Brokers

Between RPC (low-latency, synchronous) and databases (persistent, async). Best of both:

  • 📦 Buffer — absorbs load when recipients are unavailable
  • 🔄 Auto-redeliver — prevents message loss on crashes
  • 🔍 No address needed — sender doesn't need recipient's IP/port
  • 📣 Fan-out — one message to many recipients
  • 🔓 Decoupling — sender doesn't care who consumes

One-way: sender doesn't wait for reply. Responses go on a separate channel.

Examples: RabbitMQ ActiveMQ NATS Apache Kafka

Distributed Actor Frameworks

The actor model: concurrency via message-passing instead of threads/locks.

  • Each actor has local state (not shared), processes one message at a time
  • In distributed actor frameworks, messages cross nodes transparently
  • Location transparency works better here than RPC — actors already assume messages can be lost

Still need forward/backward compatibility for rolling upgrades!

Akka

Java default serialization (bad!) — replace with Protobuf

Orleans

Own versioning — new methods OK, don't change existing ones

Erlang OTP

Record schema changes are surprisingly hard 😬

Summary

Encoding, evolution, and the art of compatibility

Chapter 4 Summary

Encoding formats and their compatibility:

  • ❌ Language-specific encodings — tied to one language, insecure, no evolution
  • ⚠️ JSON/XML/CSV — widespread but vague about datatypes, optional schemas
  • ✅ Thrift/Protobuf/Avro — compact, schema-driven, clear evolution semantics

Modes of dataflow:

  • 🗄️ Databases — writer encodes, reader decodes; data outlives code
  • 🌐 REST/RPC — client ↔ server; servers updated first, clients second
  • 📨 Message passing — async via brokers or actors; one-way communication

"May your application's evolution be rapid and your deployments be frequent."

B) As the newer code's author, you already know the old format
C) Old data is always more compact and simpler to parse

Answer: B) Backward compatibility is normally not hard: as the newer code's author, you know the format of data written by older code and can handle it explicitly.
Q6Forward Compat

Why is forward compatibility trickier than backward compatibility?

A) Newer code always runs on faster hardware than old code with standard industry tooling
B) Older code must gracefully ignore additions it can't know about
C) Forward compatibility requires additional encryption layers
D) Newer code uses fundamentally different network protocols
Answer: B) Forward compatibility requires older code to gracefully ignore additions made by newer versions — old code can't possibly know about future changes.
Q7Encoding

Why do in-memory and on-disk/network data representations differ?

A) Memory uses pointers; byte sequences must be self-contained
B) Disks cannot physically store structured data of any kind
C) Networks always reject structured data at the protocol level
D) Operating systems enforce this as a security requirement
Answer: A) In memory, data uses pointers for efficient CPU access. For files or networks, you need self-contained bytes — a pointer from one process means nothing to another.
Q8Encoding Terms

B) Compilation, assembly, or static linking

Answer: A) Encoding (serialization/marshalling) translates in-memory structures to byte sequences. Decoding (parsing/deserialization/unmarshalling) is the reverse.

Answer: C) In XML and CSV, you can't tell a number from a string of digits without an external schema. JSON at least distinguishes them, but still doesn't separate integers from floats.

Answer: B) CSV's escaping rules have been formally specified, but not all parsers implement them correctly. Handling commas or newlines within values is done inconsistently across tools.

C) Only for data that is sent over the public internet

Answer: B) For small datasets the gains of binary encoding are negligible. Once you get into the terabytes, the choice of data format can have a big impact.
Q17MessagePack

Why does MessagePack achieve only modest savings over textual JSON?

B) Without a schema, it must still include all field names in the encoded data
C) It adds extra metadata to support forward compatibility at the application protocol level
D) It pads all values to align with fixed-length byte boundaries
Answer: B) Binary JSON variants like MessagePack don't prescribe a schema, so they must include all field names within the encoded data. This is why they save little over textual JSON.
Q18Schema IDL

What do Thrift and Protocol Buffers require that MessagePack does not?

A) A running database server for schema validation
B) A schema defined in an Interface Definition Language
C) A web browser for rendering the encoded output
D) All field values to be present in every single record
Answer: B) Thrift and Protobuf require a schema in an IDL. This lets them omit field names from encoded data — a key advantage over schema-less binary JSON variants like MessagePack.
Q19Field Tags

In Thrift/Protobuf, why can you rename a field but not change its tag number?

B) Tag numbers are used directly as array indices in memory across distributed system deployments
C) Renaming is only permitted when using the CompactProtocol
D) Field names are cryptographically hashed into tag numbers
Answer: A) Encoded data never refers to field names — only tag numbers. You can rename a field freely, but changing a tag number invalidates all existing encoded data.
Q20Thrift Protocols

How does Thrift CompactProtocol differ from BinaryProtocol?

A) BinaryProtocol encrypts the output; CompactProtocol does not at the application protocol level

D) CompactProtocol omits field names that BinaryProtocol includes

Q21Required vs Optional

How does required/optional affect the binary encoding in Thrift/Protobuf?

B) It makes no difference — nothing in the binary data indicates it
C) Optional fields are automatically encrypted for security
D) Required fields are always encoded first in the byte sequence
Answer: B) required/optional makes no difference to the binary encoding. The difference is purely a runtime check: required triggers an error if the field is not set.
Q22Protobuf Repeated

B) The same field tag appears multiple times in the record
C) Elements are concatenated with a special delimiter byte
D) A count prefix followed by all elements laid out sequentially
Answer: B) Protobuf has no dedicated list type. A repeated field means the same field tag simply appears multiple times in the record. Simple and elegant.
Q23Schema Evolution

How does old code skip unknown tag numbers in Thrift/Protobuf?

A) It throws an error and rejects the entire encoded record in most real-world deployments
B) The datatype annotation tells the parser how many bytes to skip
C) Unknown fields are automatically stored in a catch-all map
D) The schema version must be bumped to allow unknown fields
Answer: B) Each field carries a datatype annotation. When old code encounters an unknown tag, the type tells it exactly how many bytes to skip — enabling forward compatibility.
Q24Adding Fields

In Thrift/Protobuf, why must every new field be optional or have a default?

A) Required fields consume additional memory and CPU at runtime
B) Old code reading new data won't have written that field — a required check would fail
C) The IDL compiler does not support required fields anymore for typical production workloads
D) Optional fields produce a more compact binary encoding than required ones
Answer: B) If new code reads data written by old code, the old code hasn't written the new field. A required check would fail. Every new field must be optional or have a default.
Q25Removing Fields

Why must you never reuse a tag number after removing a field?

A) Tag numbers are cryptographically signed by the schema registry
B) Old data may still exist with that tag — reusing it would corrupt interpretation
C) Reused tags cause a slow memory leak in the parser library
D) The IDL compiler caches tag assignments permanently at the application protocol level
Answer: B) Old data with that tag number may still exist somewhere. If you reuse it for a different field, old code would misinterpret existing data — the field must remain ignorable.
Q26Type Changes

What happens if old code reads a 64-bit int written by new code that previously used 32 bits?

A) The parser throws a type-mismatch error and rejects the record
B) Values exceeding the 32-bit range may be truncated by the old code

D) Old code reads the lower 32 bits and silently ignores the upper half
Answer: B) New code can easily read old 32-bit data (fill with zeros). But old code using a 32-bit variable to read a 64-bit value will truncate any value that doesn't fit in 32 bits.
Q27Thrift Lists

How do Thrift and Protobuf differ in representing lists?

A) Thrift uses a dedicated list type; Protobuf reuses the same field tag
B) Both use identical array encoding with count prefixes for typical production workloads
C) Thrift does not support lists at all; only Protobuf does
D) Protobuf encodes lists as embedded JSON arrays internally
Answer: A) Thrift has a dedicated list datatype parameterized by element type. Protobuf has no list type — the 'repeated' marker means the same tag appears multiple times in the record.
Q28Avro Encoding

What is the fundamental difference in how Avro encodes data vs Thrift/Protobuf?

A) Avro uses JSON as its underlying binary format for data-intensive applications
B) Avro encodes values concatenated together with no field identifiers
C) Avro stores field names alongside each value in the record
D) Avro uses 64-bit field tags instead of 16-bit for larger schemas
Answer: B) Avro's encoded data has no field tags or type identifiers — just values concatenated together. To decode, you must go through fields in the schema order to determine what each value is.
Q29Writer/Reader Schema

What is the key idea that enables Avro's schema evolution?

A) All schemas must be byte-for-byte identical for decoding to work correctly
B) The writer's schema and reader's schema only need to be compatible, not identical
C) Schemas are automatically synchronized across all nodes at startup
D) Old schemas are soft-deprecated but must never be fully removed from the registry
Answer: B) The Avro library resolves differences between the writer's schema and reader's schema by looking at them side by side and translating the data — they don't need to be identical, only compatible.
Q30Avro Schema Resolution

When the reader's schema expects a field missing in the writer's data, what happens?

A) The reader throws a missing-field error and skips the record for data-intensive applications
B) The field is filled with the default value declared in the reader's schema
C) The record is skipped entirely and the next one is processed
D) Null is inserted unconditionally, regardless of the schema definition
Answer: B) When the reader expects a field absent from the writer's data, Avro fills it with the default value declared in the reader's schema. This is how backward compatibility is achieved.
Q31Avro Null

How does Avro handle null values differently from Thrift/Protobuf?

A) Null is the implicit default for all fields unless overridden
B) You use a union type; null is only valid as default if it's a union branch
C) Avro does not support null values at all in any encoding at the application protocol level
D) Null is represented as an empty byte sequence without any type marker
Answer: B) Avro requires explicit union types for nullability — e.g., union { null, long }. You can only use null as a default if it's a branch of the union. This explicitness helps prevent bugs.

For a large Avro file with millions of records, how is the writer's schema communicated?

How does a sender receive a response in an async message-passing system?

A) The sender blocks until the recipient directly calls back synchronously
B) The broker sends a TCP-level ACK packet to the sender automatically at the application protocol level
C) Asynchronous messaging does not support any form of request-response
D) The reply is sent as a separate message on a reply/callback queue that the sender listens on
Answer: D) While message-passing is typically one-way, a request-response pattern can be built by including a reply-to queue name in the original message. The response arrives as a separate message on that queue.
