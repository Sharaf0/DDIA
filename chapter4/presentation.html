Formats for Encoding Data

From in-memory objects to bytes on the wire

Two Representations of Data

🧠 In Memory

Objects, structs, lists, hash tables, trees…

Optimized for CPU access via pointers

📡 On the Wire / Disk

Self-contained sequence of bytes

Pointers are meaningless to other processes

Translation: Encoding (serialization/marshalling) ↔ Decoding (parsing/deserialization/unmarshalling)

Note: the book prefers “encoding” because “serialization” also means something different in transactions (Chapter 7).

Language-Specific Formats ❌

Java’s Serializable, Ruby’s Marshal, Python’s pickle, Kryo…

  • Tied to one language — can’t read the data from another language
  • Security risk — decoding can instantiate arbitrary classes → remote code execution 💀
  • Versioning is an afterthought — poor forward/backward compatibility
  • Poor efficiency — Java serialization is notorious for bloated encoding

“It’s generally a bad idea to use your language’s built-in encoding for anything other than very transient purposes.”

JSON, XML, and CSV

Widely known, widely supported, and almost as widely disliked. 😅

  • Number ambiguity — JSON can’t distinguish integers from floats; integers > 2⁵³ break in JavaScript (Twitter includes tweet IDs twice!)
  • No binary strings — must Base64 encode → 33% size increase
  • Schema support — optional for XML/JSON, nonexistent for CSV
  • CSV quirks — commas in values? Newlines? Escaping rules aren’t always followed

Despite flaws, they’re good enough for data interchange between organizations. The difficulty of getting different organizations to agree on anything outweighs most other concerns.

Binary Encoding: MessagePack

JSON/XML are verbose → binary variants exist: MessagePack BSON BJSON Smile

Example record:

{ "userName": "Martin", "favoriteNumber": 1337, "interests": ["daydreaming", "hacking"] }

MessagePack: 66 bytes vs JSON text: 81 bytes

Only ~18% saving. Still includes field names. We can do much better…

Thrift & Protocol Buffers

Schema-driven binary encoding from Facebook & Google

Schema Definitions

Thrift IDL

struct Person { 1: required string userName, 2: optional i64 favoriteNumber, 3: optional list<string> interests }

Protocol Buffers

message Person { required string user_name = 1; optional int64 favorite_number = 2; repeated string interests = 3; }

Both use numbered field tags instead of field names → much more compact encoding

Thrift BinaryProtocol

Each field encoded with:

  • Type annotation (string, integer, list…)
  • Field tag number (1, 2, 3) instead of names
  • Length indication where needed
59 bytes
vs 66 for MessagePack, 81 for JSON text

Thrift CompactProtocol

Packs field type + tag into a single byte

Uses variable-length integers: numbers −64 to 63 fit in 1 byte, −8192 to 8191 in 2 bytes

The number 1337 is encoded in just 2 bytes instead of a full 8.

34 bytes
Same data, ~half the size of BinaryProtocol

Protocol Buffers Encoding

Similar bit packing to Thrift CompactProtocol, slightly different approach.

33 bytes
Most compact of Thrift/Protobuf encodings

required vs optional makes no difference in the binary encoding — nothing in the encoded bytes indicates it. It’s only a runtime validation check.

Field Tags & Schema Evolution

Field tags are critical — they’re how encoded data identifies fields.

✅ You CAN

  • Change field names (tags stay the same)
  • Add new fields with new tag numbers
  • Remove optional fields

❌ You CANNOT

  • Change a field’s tag number
  • Make a new field required
  • Remove a required field
  • Reuse a deleted field’s tag number

How Compatibility Works

→ Forward Compatibility

Old code reads new data

Old code sees unknown tag numbers → skips them (datatype annotation tells how many bytes to skip)

← Backward Compatibility

New code reads old data

New code knows tag meanings. New fields must have default values since old data won’t have them.

New fields must be optional or have a default — otherwise a required check would fail when new code reads old data.

Datatypes & Schema Evolution

Changing a field’s datatype is possible but risky:

32-bit → 64-bit Integer

  • New code reads old data: fills missing bits with zeros ✅
  • Old code reads new data: truncated if value exceeds 32 bits ⚠️

Lists: Thrift vs Protobuf

  • Protobuf: no list type — repeated = same tag appears multiple times. optional → repeated is safe (old code sees last element)
  • Thrift: dedicated list type, supports nested lists but no single → multi evolution

Apache Avro

Born from Hadoop, designed for dynamically generated schemas

Avro Schema — No Tag Numbers!

Avro IDL

record Person { string userName; union { null, long } favoriteNumber = null; array<string> interests; }

JSON Representation

{ "type": "record", "name": "Person", "fields": [ {"name": "userName", "type": "string"}, {"name": "favoriteNumber", "type": ["null","long"], "default": null}, {"name": "interests", "type": {"type":"array","items":"string"}} ] }

Two schema languages: Avro IDL for humans, JSON for machines.

Avro Encoding

32 bytes
Most compact encoding we’ve seen!

No field identifiers or type annotations in the data. Just values concatenated together.

Decoded by walking through fields in schema order. Reader MUST use the exact same schema… or must they? 🤔

Writer’s Schema vs Reader’s Schema

The key idea: they don’t have to be the same — just compatible!

✍️ Writer’s Schema

The schema used when the data was encoded. Compiled into the writing application.

📖 Reader’s Schema

The schema the reading application expects. May be a different version!

Avro resolves differences by matching fields by name, filling in defaults for missing fields, and ignoring unknown ones.

Schema Resolution in Action

  • Fields matched by name (not by position or tag number)
  • Different field order? No problem
  • Writer has field reader doesn’t? Ignored
  • Reader expects field writer doesn’t have? Filled with default

Avro Schema Evolution Rules

  • You may only add or remove a field that has a default value
  • Adding a field without a default → breaks backward compatibility
  • Removing a field without a default → breaks forward compatibility

🚫 Nullability via Union Types

Avro has no optional/required markers. To allow null, use a union type:

union { null, long, string } field;

Null is only a valid default if it’s a branch of the union. More verbose, but prevents bugs.

🔄 Other Changes

  • Datatype change: possible if Avro can convert the type
  • Field rename: reader schema can declare aliases for old names (backward-compatible, not forward-compatible)
  • Union branch added: backward-compatible but not forward-compatible

How Does the Reader Find the Writer’s Schema?

📁 Large Files

Hadoop-style: millions of records, same schema. Include schema once at file beginning.

→ Avro object container files

🗄️ Database Records

Different records may use different schemas. Include a version number per record.

→ Look up schema from version DB (e.g., Espresso)

🌐 Network

Two processes negotiate schema version on connection setup.

→ Avro RPC protocol

A database of schema versions is useful in any case — it acts as documentation and lets you check compatibility before deployment.

Avro’s Superpower: Dynamically Generated Schemas

Want to dump a relational database to a binary file?

  • Generate an Avro schema from the database schema automatically
  • Each table → a record, each column → a field
  • Column name = field name (no manual tag assignment needed!)
  • Database schema changes? Just regenerate the Avro schema. No manual intervention.

With Thrift/Protobuf, you’d need to manually maintain tag-to-column mappings — “this kind of dynamically generated schema simply wasn’t a design goal.”

Code Generation & Dynamic Languages

Thrift / Protobuf

Rely on code generation from the schema.

Useful in statically typed languages (Java, C++, C#): compile-time type checking, IDE autocompletion.

Less useful in dynamic languages — no compiler to satisfy.

Avro

Optional code generation.

Avro object container files are self-describing — just open and inspect, like a JSON file.

Especially useful with tools like Apache Pig: open Avro files, analyze, write output — without thinking about schemas.

Historical Context: ASN.1

These encoding ideas are not new:

  • ASN.1 (Abstract Syntax Notation One) — first standardized in 1984
  • Used to define network protocols; its binary encoding (DER) still encodes SSL/TLS certificates (X.509) today
  • Supports schema evolution via tag numbers, similar to Protobuf and Thrift
  • However: very complex and badly documented — not a good choice for new applications

Most relational databases also use proprietary binary encodings for their network protocols (decoded by ODBC/JDBC drivers).

The Merits of Schemas

Schema-driven binary encodings win on multiple fronts:

  • 📦 More compact — omit field names from encoded data
  • 📝 Living documentation — schema is required for decoding, so it’s always up to date
  • ✅ Compatibility checking — database of schemas lets you verify compatibility before deployment
  • 🔧 Code generation — for statically typed languages: type checking at compile time

“Schema evolution allows the same kind of flexibility as schemaless/schema-on-read JSON databases provide, while also providing better guarantees about your data and better tooling.”

Modes of Dataflow

Where and how encoded data travels between processes

Three Ways Data Flows

🗄️ Through Databases

Writer encodes, reader decodes. “Sending a message to your future self.”

🌐 Through Services

REST, RPC — client encodes request, server decodes and responds.

📨 Through Messages

Async message passing via brokers — sender encodes, consumer decodes.

Dataflow Through Databases

Backward compatibility is clearly needed — your future self must decode what you wrote.

Forward compatibility is also needed — during rolling upgrades, older code may read data written by newer code.

⚠️ Danger: Old code reads a record with a new field, updates it, and writes it back — the new field might be lost!

Encoding formats handle unknown field preservation, but watch out at the application level (model objects may drop unknown fields).

Data Outlives Code

“Within a single database you may have some values that were written five milliseconds ago, and some values that were written five years ago.”

You deploy new code in minutes. But old data stays forever in its original encoding.

  • Rewriting (migrating) all data is expensive for large datasets
  • Most relational DBs: ALTER TABLE ADD COLUMN with NULL default — no rewrite needed
  • LinkedIn’s Espresso uses Avro for storage → Avro schema evolution handles it

Schema evolution lets the DB appear as if all data uses one schema, even if it contains records from many eras.

Archival Storage

Database snapshots for backup or data warehousing:

  • Dump is encoded with the latest schema — a consistent snapshot
  • Written once, immutable — perfect for Avro object container files
  • Good opportunity to encode in an analytics-friendly format like Parquet (column-oriented)

Dataflow Through Services

REST, SOAP, RPC — and the pitfalls of location transparency

Services and Microservices

Servers expose an API (service); clients connect and make requests.

  • A server can be a client to another service — this is the foundation of microservices
  • Services provide encapsulation — restrict what clients can do via the API (unlike databases which allow arbitrary queries)
  • Key goal: services are independently deployable and evolvable

Old and new versions of servers and clients run simultaneously → data encoding must be compatible across versions.

Web Services: Three Contexts

When HTTP is the underlying protocol, it’s called a web service:

📱 Client → Server

Mobile apps, JS web apps (Ajax) making requests over the public internet.

🏢 Service → Service

Same organization, same datacenter. Part of a microservices architecture.

(Sometimes called middleware)

🌐 Org → Org

Cross-organizational data exchange: public APIs, credit card processing, OAuth.

REST vs SOAP

REST ✅

  • Design philosophy built on HTTP principles
  • Simple data formats (JSON), URLs for resources
  • Uses HTTP features: caching, auth, content negotiation
  • API described with OpenAPI/Swagger
  • Predominant for public APIs

SOAP ⚠️

  • XML-based protocol, avoids HTTP features
  • Complex standards: WS-* framework
  • WSDL for API description → code generation
  • WSDL not human-readable; heavy IDE dependence
  • Vendor interop problems; still used in large enterprises

The Problems with RPC

RPC tries to make a network call look like a local function call (location transparency). But it’s fundamentally flawed:

  • 🎲 Unpredictable — network requests can be lost, slow, or unavailable
  • ⏱️ Timeouts — no response ≠ failure. You just don’t know what happened
  • 🔄 Retries need idempotence — the request may have succeeded; only the response was lost
  • 🐌 Wildly variable latency — sub-millisecond to multi-second for the same call
  • 📦 No pointer passing — all parameters must be serialized to bytes
  • 🌐 Cross-language type mismatches — e.g., JS can’t handle integers > 2⁵³

Modern RPC Frameworks

RPC isn’t going away — but new frameworks are more honest about network realities:

  • gRPC Protocol Buffers, supports streams (series of requests/responses over time)
  • Finagle Rest.li Use futures/promises for async + parallel requests
  • Avro RPC Schema negotiation on connection setup

Some frameworks also provide service discovery — finding which IP:port hosts a particular service.

REST wins for public APIs (easy debugging with curl/browser, vast ecosystem). RPC wins for internal services (performance, efficiency).

Evolution for RPC

Simplifying assumption: servers updated first, clients second.

  • Need backward compatibility on requests (new server reads old client’s data)
  • Need forward compatibility on responses (old client reads new server’s data)

Compatibility rules depend on the encoding format used (Thrift, Protobuf, Avro, JSON, XML schemas…)

Cross-organization APIs often require indefinite compatibility — the provider can’t force clients to upgrade. Common to maintain multiple API versions side by side (version in URL, Accept header, or per-client API key).

Message-Passing Dataflow

Async communication via message brokers and actors

Message Brokers

Between RPC (low-latency, synchronous) and databases (persistent, async). Best of both:

  • 📦 Buffer — absorbs load when recipients are unavailable
  • 🔄 Auto-redeliver — prevents message loss on crashes
  • 🔍 No address needed — sender doesn’t need recipient’s IP/port
  • 📣 Fan-out — one message to many recipients
  • 🔓 Decoupling — sender doesn’t care who consumes

One-way: sender doesn’t wait for reply. Responses go on a separate channel.

Examples: RabbitMQ ActiveMQ HornetQ NATS Apache Kafka

Message Broker Internals

One process sends a message to a named queue or topic; the broker delivers to consumers/subscribers.

  • A topic provides one-way dataflow — but consumers can republish to other topics (chaining)
  • A reply queue can simulate request/response (similar to RPC)
  • Brokers don’t enforce a data model — a message is just bytes + metadata
  • Use a backward/forward compatible encoding for maximum deployment flexibility

⚠️ If a consumer republishes messages to another topic, be careful to preserve unknown fields — same issue as the database scenario (Figure 4-7).

Distributed Actor Frameworks

The actor model: concurrency via message-passing instead of threads/locks.

  • Each actor has local state (not shared), processes one message at a time
  • In distributed actor frameworks, messages cross nodes transparently
  • Location transparency works better here than RPC — actors already assume messages can be lost

Still need forward/backward compatibility for rolling upgrades!

Akka

Java default serialization (bad!) — replace with Protobuf for rolling upgrades

Orleans

Own versioning mechanism: add new actor methods, keep existing ones unchanged

Erlang OTP

Record schema changes are surprisingly hard, despite high-availability design 😬

Summary

Encoding, evolution, and the art of compatibility

Chapter 4 Summary

Encoding formats and their compatibility:

  • ❌ Language-specific encodings — tied to one language, insecure, no evolution
  • ⚠️ JSON/XML/CSV — widespread but vague about datatypes, optional schemas
  • ✅ Thrift/Protobuf/Avro — compact, schema-driven, clear evolution semantics via field tags or name matching

Modes of dataflow:

  • 🗄️ Databases — writer encodes, reader decodes; data outlives code; preserve unknown fields
  • 🌐 REST/RPC — client ↔ server; servers updated first, clients second; indefinite compatibility for public APIs
  • 📨 Message passing — async via brokers or actors; one-way communication; encoding flexibility

“May your application’s evolution be rapid and your deployments be frequent.”

B) How easily a system adapts to changing requirements
C) The mean time between hardware-induced failures

Answer: B) Evolvability is the ease with which you can modify a data system and adapt it to changing requirements, introduced in Chapter 1.
Q3Rolling Upgrades

What is the primary benefit of a rolling upgrade?

A) It reduces the disk space used by the application binary
B) It deploys new versions without service downtime
C) It automatically fixes bugs in the previous version
D) It eliminates the need for backward compatibility
Answer: B) Rolling upgrades deploy to a few nodes at a time, checking each before proceeding. This means no downtime and enables more frequent, lower-risk releases.
Q4Backward Compat

Why is backward compatibility usually easier than forward compatibility?

A) Old data formats are always formally documented
B) As the newer code's author, you already know the old format
C) Old data is always simpler and more compact to parse
D) Newer programming languages have superior deserialization APIs
Answer: B) Backward compatibility is normally not hard: as the newer code's author, you know the format of data written by older code and can handle it explicitly.
Q5Forward Compat

Why is forward compatibility trickier than backward compatibility?

A) Newer code always runs on faster hardware
B) Older code must gracefully ignore additions it can't know about
C) Forward compatibility requires additional encryption layers
D) Newer code uses fundamentally different network protocols
Answer: B) Forward compatibility requires older code to gracefully ignore additions made by newer versions — old code can’t possibly know about future changes.
Q6Encoding

Why do in-memory and on-disk/network data representations differ?

A) Memory uses pointers; byte sequences must be self-contained
B) Disks cannot physically store structured data
C) Networks reject structured data at the transport layer
D) Operating systems enforce separate formats as a security requirement
Answer: A) In memory, data uses pointers for efficient CPU access. For files or networks, you need self-contained bytes — a pointer from one process means nothing to another.
Q7Encoding Terms

What is the translation from in-memory to byte sequence called?

A) Encoding, serialization, or marshalling
B) Compilation, assembly, or static linking
C) Encryption, hashing, or digital signing
D) Indexing, partitioning, or sharding
Answer: A) Encoding (serialization/marshalling) translates in-memory structures to byte sequences. Decoding (parsing/deserialization/unmarshalling) is the reverse.

B) "Serialization" is also used in transactions with a different meaning
C) "Encoding" is the official IEEE standard term
D) "Serialization" only applies to the Java programming language
Answer: B) “Serialization” is also used in the context of transactions (Chapter 7) with a completely different meaning. The book sticks with “encoding” to avoid confusion.
Q9Language Formats

Which is NOT a problem with language-specific encoding libraries?

A) The encoding ties you to a particular programming language
B) Arbitrary class instantiation during decoding is a security risk
C) Versioning and compatibility are often afterthoughts
D) They are too complex, requiring extensive configuration
Answer: D) Language-specific encodings are actually very convenient with minimal code. Their real problems: language lock-in, security risks, neglected versioning, and poor efficiency.

Why is Java's built-in serialization considered especially problematic?

Answer: B) Java’s built-in serialization is notorious for bad performance and bloated encoding. Language built-in encodings should only be used for very transient purposes.
Q11JSON Numbers

Why does Twitter's API return tweet IDs as both a number and a string?

A) For backward compatibility with legacy XML parsers
B) Integers beyond 2^53 can't be exactly represented in IEEE 754 doubles
C) To reduce bandwidth by offering two different precision levels
D) The JSON specification requires all numbers to be duplicated
Answer: B) JSON doesn’t distinguish ints from floats, using IEEE 754 doubles. Integers > 2^53 lose precision. Twitter returns IDs as both number and string so JavaScript clients get accurate values.
Q12JSON Binary

How do people work around JSON's lack of native binary string support?

A) Binary data is stored in a separate companion file
B) Encode binary as text with Base64, increasing size by about 33%
C) Convert binary to hexadecimal digits within JSON strings
D) Switch to XML which natively supports binary payloads
Answer: B) JSON and XML lack native binary string support. The workaround: Base64-encode binary data, then use a schema to indicate it should be decoded. Adds ~33% overhead.
Q13XML/CSV Types

What type ambiguity do XML and CSV share?

A) Neither format can represent nested, hierarchical data
B) Both formats only support unsigned integer types
C) A number and a digit string are indistinguishable without a schema
D) Neither format supports Unicode character encoding
Answer: C) In XML and CSV, you can’t tell a number from a string of digits without an external schema. JSON at least distinguishes them, but still doesn’t separate integers from floats.
Q14CSV

What makes CSV a particularly vague format?

A) It cannot represent floating-point numbers
B) Escaping rules for commas and newlines vary across parsers
C) It requires a proprietary parser to read
D) It only supports 7-bit ASCII, not Unicode
Answer: B) CSV’s escaping rules have been formally specified, but not all parsers implement them correctly. Handling commas or newlines within values is done inconsistently.
Q15Binary Encoding

When does the choice of binary vs textual format become important?

A) For all data — binary should always be the default
B) At the terabyte scale the format choice has a big impact
C) Only for data that crosses organizational boundaries
D) Only when using dynamically typed languages
Answer: B) For small datasets the gains of binary encoding are negligible. Once you get into the terabytes, the choice of data format can have a big impact.
Q16MessagePack

Why does MessagePack achieve only modest savings over textual JSON?

A) It uses an inefficient compression algorithm
B) Without a schema, it must still include all field names in the encoded data
C) It adds extra metadata to support schema evolution
D) It pads all values to align with 8-byte boundaries
Answer: B) Binary JSON variants like MessagePack don’t prescribe a schema, so they must include all field names within the encoded data. This is why they save little over textual JSON.
Q17Schema IDL

What do Thrift and Protocol Buffers require that MessagePack does not?

A) A running database server for schema validation
B) A schema defined in an Interface Definition Language
C) A web browser for rendering the encoded output
D) All field values to be present in every record
Answer: B) Thrift and Protobuf require a schema in an IDL. This lets them omit field names from encoded data — a key advantage over schema-less binary JSON variants.
Q18Field Tags

In Thrift/Protobuf, why can you rename a field but not change its tag number?

A) Field names are in the schema only; the encoded data uses tag numbers
B) Tag numbers are used as array indices in the runtime memory layout
C) Renaming is only permitted when using CompactProtocol
D) Field names are cryptographically hashed into tag numbers
Answer: A) Encoded data never refers to field names — only tag numbers. You can rename a field freely, but changing a tag number invalidates all existing encoded data.
Q19Thrift Protocols

How does Thrift CompactProtocol differ from BinaryProtocol?

A) BinaryProtocol encrypts the output; CompactProtocol does not
B) CompactProtocol packs field type and tag into one byte, uses varints
C) BinaryProtocol targets files while CompactProtocol targets networks
D) CompactProtocol omits field names that BinaryProtocol includes
Answer: B) CompactProtocol packs field type and tag into a single byte and uses variable-length integers. Result: 34 bytes vs BinaryProtocol’s 59 bytes for the same record.
Q20Required vs Optional

How does required/optional affect the binary encoding in Thrift/Protobuf?

A) Required fields always use fewer bytes in the output
B) It makes no difference — nothing in the binary data indicates it
C) Optional fields include a presence flag byte before the value
D) Required fields are always encoded first in the byte sequence
Answer: B) required/optional makes no difference to the binary encoding. The difference is purely a runtime check: required triggers an error if the field is not set.
Q21Adding Fields

What must every new field added to a Thrift/Protobuf schema have?

A) A unique tag number not used by any current or previous field
B) A default value defined in the encoding specification
C) An accompanying migration script
D) A type identical to the field directly preceding it
Answer: A) Every new field must have a new, unique tag number. Old code will see the unknown tag and skip it (forward compatibility). You can never reuse a deleted tag.
Q22Required Field

Why can't you make a new field "required" when adding it to a Protobuf schema?

A) Protobuf only allows optional fields after version 3
B) Old code won't write it, so new code reading old data would fail the check
C) Required fields are not allowed in any version of Protobuf
D) New fields are automatically treated as repeated
Answer: B) If you add a required field, old code (which doesn’t know about it) won’t write it. New code reading old data will fail the runtime check — breaking backward compatibility.
Q23Removing Fields

What rule applies when removing a field from a Thrift/Protobuf schema?

A) Required fields may be removed freely since old data is migrated
B) Only optional fields can be removed, and their tag must never be reused
C) Fields can only be removed if no data has been written using them
D) Any field can be removed as long as its tag is assigned to a new field
Answer: B) You can only remove optional fields. The tag number must never be reused — old data still contains that tag, and future code must ignore it.
Q24Datatypes

What risk arises when changing a field from int32 to int64 in Protobuf?

A) Old code reading new data silently truncates the 64-bit value to 32 bits
B) The encoder will refuse to serialize any value over 2^31
C) Old and new code produce different tag numbers for the same field
D) The field type byte changes and all existing data becomes unreadable
Answer: A) Changing int32 → int64 is fine for new code reading old data (values fit). But old code reading new 64-bit data may truncate it to 32 bits, losing precision.
Q25Proto Repeated

How does Protobuf handle changing an optional single-valued field to a repeated field?

A) It can’t — the change requires creating a new field tag
B) New code sees old single values as a list with one element; old code reads only the last element
C) Both old and new code can only see the first element of any list
D) A migration automatically converts single values to arrays in all records
Answer: B) In Protobuf, repeated fields are just multiple tag/value pairs. New code treats one value as a length-1 list; old code seeing many values takes the last one.
Q26Thrift Lists

Why can't Thrift evolve an optional field to a list the way Protobuf can?

A) Thrift encodes lists with a dedicated list type; it’s not just repeated tags
B) Thrift does not support list data types at all
C) Thrift uses JSON internally which cannot represent lists
D) Thrift requires a separate schema version for every list field
Answer: A) Thrift has a dedicated list datatype with nested encoding. This allows nested lists (lists of lists) which Protobuf can’t do, but prevents evolving a single value into a list.
Q27Avro

What makes Avro's binary encoding the most compact among Thrift, Protobuf, and Avro?

A) It uses stronger compression like LZ4 on each record
B) It has no tag numbers or field identifiers in the byte stream at all
C) It stores only the difference between each record and the previous one
D) It converts all text to 7-bit ASCII before encoding
Answer: B) Avro has no tag numbers or field type indicators in the encoding. Values are simply concatenated in the order the schema defines. This is why it’s the most compact.
Q28Avro Decoding

How does an Avro decoder know the type and meaning of each byte in the stream?

A) Each value is preceded by a type tag byte
B) A schema must be available — bytes are interpreted in schema field order
C) Self-describing headers are embedded at the start of each field
D) Values are length-prefixed so the decoder can count bytes
Answer: B) An Avro decoder must have the exact schema that was used to write the data. It goes through fields in order, using the schema to know each field’s datatype and length.
Q29Writer/Reader Schema

In Avro, what is the difference between the writer's schema and the reader's schema?
