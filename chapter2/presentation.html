Data models are perhaps the most important part of developing software — they shape not only how the software is written, but how we think about the problem.

Layers of Data Models

Every application is built by layering models:

  • Layer 1: Real world → objects, data structures, APIs
  • Layer 2: Objects → JSON, XML, tables, graphs
  • Layer 3: Data model → bytes on disk/memory
  • Layer 4: Bytes → electrical currents, magnetic fields

Each layer hides the complexity below it by providing a clean abstraction.

Relational vs Document

The great data model debate

The Relational Model

Proposed by Edgar Codd in 1970.

  • Data organized into relations (tables)
  • Each relation is an unordered collection of tuples (rows)
  • Dominant for 25–30 years — an eternity in computing

Originally for business data processing on mainframes: transactions, invoicing, payroll.

The Relational Reign

Many competitors rose and fell. SQL kept winning:

1970s
Network & Hierarchical models
1980s
Relational model dominates
1990s
Object databases rise & fall
2000s
XML databases — niche only
2010s
NoSQL arrives

Relational databases generalized far beyond their original scope. Most of the web still runs on them.

The Birth of NoSQL

Originally just a catchy Twitter hashtag for a 2009 meetup. Later reinterpreted as Not Only SQL.

Driving forces:

  • Need for greater scalability (huge datasets, high write throughput)
  • Preference for free and open source over commercial DBs
  • Specialized query operations not well supported by relational
  • Frustration with rigid schemas

Polyglot Persistence

Different applications have different requirements.

Relational databases will continue alongside nonrelational datastores — the best choice depends on the use case.

🔀
Use the right tool for the right job

The Object-Relational Mismatch

Most apps use OOP, but SQL stores data in tables.

This disconnect is called impedance mismatch — like mismatched electrical circuits.

ORM frameworks (ActiveRecord, Hibernate) help but can't fully hide the differences.

Example: A LinkedIn Profile

How to represent a résumé in SQL?

  • users table: first_name, last_name (1 per user)
  • positions table: one-to-many via foreign key
  • education table: one-to-many via foreign key
  • contact_info table: one-to-many via foreign key

Multiple tables, multiple joins to load one profile.

The JSON Alternative

A résumé is a self-contained document. JSON feels more natural:

{ "user_id": 251, "first_name": "Bill", "positions": [ {"job_title": "Co-chair", "org": "Gates Foundation"} ], "education": [ {"school": "Harvard", "start": 1973} ] }

Better locality — one query fetches everything.

Document = Tree Structure

One-to-many relationships naturally form a tree:

  • User → positions
  • User → education
  • User → contact info

JSON makes this tree structure explicit.

Document-oriented DBs (MongoDB, CouchDB, RethinkDB) support this model natively.

Many-to-One: Why Use IDs?

Why store region_id instead of "Greater Seattle Area"?

  • Consistent style and spelling
  • No ambiguity (same-name cities)
  • Easy updates — change name in one place
  • Localization — display in viewer's language
  • Better search — Seattle is in Washington

This is the key idea behind normalization — remove duplication.

Many-to-One Doesn't Fit Documents

Normalization requires many-to-one relationships (many people → one region).

In relational DBs: easy — just join by ID.

In document DBs: joins are often weak or absent.

Without DB joins

You emulate joins in application code — multiple queries, more complexity, worse performance.

Data gets interconnected

Features grow → more relationships → document model becomes awkward.

Growing Interconnections

As features are added, data becomes more linked:

  • Organizations as entities — not just strings, but links with logos, news feeds
  • Recommendations — reference to another user's profile and photo

What started as a simple document now needs many-to-many relationships.

Many-to-Many Relationships

References to organizations, schools, and other users require joins when queried.

The data within each "document" can be grouped, but the references between documents are unavoidable.

This is where the document model starts to struggle.

Are We Repeating History?

From IMS to CODASYL to SQL to NoSQL

The Hierarchical Model (1968)

IBM's IMS — originally for Apollo space program stock-keeping. Still running today on mainframes!

  • Data stored as a tree of records
  • Remarkably similar to JSON/document model
  • Great for one-to-many
  • Terrible for many-to-many and joins

The exact same problems developers face with document DBs today.

Two Solutions Emerged

🕸️ Network Model (CODASYL)

  • Records could have multiple parents
  • Links between records = pointers on disk
  • Navigate via access paths
  • Like traversing an n-dimensional data space
  • Code was complicated and inflexible

📊 Relational Model (SQL)

  • Data laid out in the open — tables and rows
  • No labyrinthine nested structures
  • Query optimizer chooses access paths automatically
  • Just declare a new index → auto-used
  • Much easier to add new features

The relational model won because the query optimizer was a one-time investment that benefited all applications.

Document DBs: Back to Hierarchical?

Document DBs reverted to storing nested records within their parent — like the hierarchical model.

But they didn't follow CODASYL's path:

  • Related items referenced by unique identifiers (foreign keys / document references)
  • Identifiers resolved at read time via joins or follow-up queries
  • No manual access path navigation

Relational and document models are not fundamentally different in how they handle many-to-one/many-to-many.

Relational vs Document Today

Tradeoffs in the data model

Which Model → Simpler Code?

📄 Use Document If...

  • Data has a tree structure (one-to-many)
  • Entire tree loaded at once
  • Avoids "shredding" into many tables
  • Few or no many-to-many relationships

📊 Use Relational If...

  • Data has many-to-many relationships
  • Joins are needed frequently
  • Data is highly interconnected
  • Denormalization would be too costly

For highly interconnected data: document is awkward, relational is OK, graph is best.

Schema-on-Read vs Schema-on-Write

Document DBs are sometimes called "schemaless" — misleading!

Schema-on-read: structure is implicit, interpreted when data is read. Like dynamic typing.

Schema-on-write: schema is explicit, enforced on write. Like static typing.

Neither is universally better — it depends on whether your data is homogeneous or heterogeneous.

Changing Data Format

Want to split name into first_name and last_name?

📄 Document DB

// Handle old documents at read time if (user && user.name && !user.first_name) { user.first_name = user.name.split(" ")[0]; }

📊 Relational DB

ALTER TABLE users ADD COLUMN first_name text; UPDATE users SET first_name = split_part(name, ' ', 1);

Most DBs run ALTER TABLE in milliseconds. MySQL is the exception — it copies the entire table. 😬

Data Locality for Queries

A document is stored as a single continuous string (JSON, BSON, etc.).

If you often load the entire document → locality advantage: one read, no joins.

But:

  • DB must load the entire document even for partial access
  • Updates rewrite the whole document
  • Keep documents small

Locality isn't exclusive to documents: Google Spanner, Oracle, Cassandra offer similar features in relational/column-family models.

Convergence: The Best of Both

The two models are growing closer:

  • Relational → Document features: PostgreSQL, MySQL, DB2 now support JSON natively
  • Document → Relational features: RethinkDB supports joins; MongoDB resolves document references

A hybrid of relational and document is the likely future.

The data models complement each other.

Query Languages for Data

Declarative vs Imperative

Imperative vs Declarative

⚙️ Imperative

Tell the computer how to do it, step by step:

function getSharks() { var sharks = []; for (var i = 0; i < animals.length; i++) { if (animals[i].family === "Sharks") { sharks.push(animals[i]); } } return sharks; }

✨ Declarative (SQL)

Tell the computer what you want:

SELECT * FROM animals WHERE family = 'Sharks';

The DB decides how to execute it.

Why Declarative Wins

  • More concise and easier to work with
  • Hides implementation details — DB can optimize behind the scenes
  • Order of data doesn't matter — DB can move records around freely
  • Parallelizable: declarative only specifies the result pattern, not the algorithm, so the DB can parallelize execution

CPUs get faster by adding more cores, not higher clock speeds. Imperative code specifies a strict order that's hard to parallelize.

Declarative on the Web

The same principle applies in browsers!

/* CSS — Declarative */ li.selected > p { background-color: blue; }

vs 15+ lines of imperative JavaScript with getElementsByTagName, loops, and manual style setting.

CSS: auto-updates when class changes. JS: stays blue until page reload. 🤦

MapReduce Querying

A programming model for bulk data processing across many machines. Popularized by Google.

Neither fully declarative nor imperative — somewhere in between.

Logic expressed as map() and reduce() functions that must be pure (no side effects).

Used by MongoDB and CouchDB for read-only queries across documents.

MapReduce: Counting Sharks

SQL

SELECT date_trunc('month', timestamp) AS month, sum(num_animals) AS total FROM observations WHERE family = 'Sharks' GROUP BY month;

MongoDB MapReduce

db.observations.mapReduce( function map() { var y = this.ts.getFullYear(); var m = this.ts.getMonth() + 1; emit(y + "-" + m, this.numAnimals); }, function reduce(key, values) { return Array.sum(values); }, { query: {family: "Sharks"} } );

MongoDB later added the aggregation pipeline — a declarative alternative that "reinvents SQL in disguise." 😄

MongoDB Aggregation Pipeline

db.observations.aggregate([ { $match: { family: "Sharks" } }, { $group: { _id: { year: { $year: "$observationTimestamp" }, month: { $month: "$observationTimestamp" } }, totalAnimals: { $sum: "$numAnimals" } } } ]);

JSON-based syntax instead of SQL's English-style — the difference is a matter of taste.

The moral: NoSQL systems often end up reinventing SQL. 🔄

Graph-Like Data Models

When everything is connected to everything

When Graphs Shine

If many-to-many relationships are the norm, think graphs.

A graph = vertices (nodes) + edges (relationships).

Classic examples:

  • Social graphs — people who know each other
  • The web — pages linked by hyperlinks
  • Road networks — junctions and roads

Graphs aren't limited to homogeneous data — Facebook stores people, locations, events, comments in one graph.

Example: Lucy & Alain

Two people with different regional structures:

  • Lucy: born in Idaho, USA (counties → states → country)
  • Alain: from Beaune, France (départements → régions → country)
  • Both living in London

Different countries have different administrative hierarchies — graphs handle this naturally.

Easily extensible: add allergies, food contents, etc.

Property Graph Model

Used by Neo4j, Titan, InfiniteGraph.

Vertex

  • Unique identifier
  • Set of outgoing edges
  • Set of incoming edges
  • Properties (key-value pairs)

Edge

  • Unique identifier
  • Tail vertex (start)
  • Head vertex (end)
  • Label (relationship type)
  • Properties (key-value pairs)

Key insight: any vertex can connect to any other vertex. No schema restricts edge types.

Property Graph in SQL

CREATE TABLE vertices ( vertex_id integer PRIMARY KEY, properties json ); CREATE TABLE edges ( edge_id integer PRIMARY KEY, tail_vertex integer REFERENCES vertices (vertex_id), head_vertex integer REFERENCES vertices (vertex_id), label text, properties json ); CREATE INDEX edges_tails ON edges (tail_vertex); CREATE INDEX edges_heads ON edges (head_vertex);

You can store graph data in a relational DB. But querying it is... verbose.

Cypher: Graph Query Language

Declarative query language created for Neo4j. Named after a Matrix character, not cryptography. 😎

CREATE (NAmerica:Location {name:'North America', type:'continent'}), (USA:Location {name:'United States', type:'country'}), (Idaho:Location {name:'Idaho', type:'state'}), (Lucy:Person {name:'Lucy'}), (Idaho) -[:WITHIN]-> (USA) -[:WITHIN]-> (NAmerica), (Lucy) -[:BORN_IN]-> (Idaho)

Arrow notation: (A) -[:LABEL]-> (B)

Cypher: Find Emigrants

"Find people who emigrated from the US to Europe":

MATCH (person) -[:BORN_IN]-> () -[:WITHIN*0..]-> (us:Location {name:'United States'}), (person) -[:LIVES_IN]-> () -[:WITHIN*0..]-> (eu:Location {name:'Europe'}) RETURN person.name

:WITHIN*0.. = follow WITHIN edges zero or more times (like regex *).

The query optimizer decides whether to start from people and filter, or start from locations and work backwards.

Same Query in SQL: 29 Lines 😱

WITH RECURSIVE in_usa(vertex_id) AS ( SELECT vertex_id FROM vertices WHERE properties->>'name' = 'United States' UNION SELECT edges.tail_vertex FROM edges JOIN in_usa ON edges.head_vertex = in_usa.vertex_id WHERE edges.label = 'within' ), in_europe(vertex_id) AS ( SELECT vertex_id FROM vertices WHERE properties->>'name' = 'Europe' UNION SELECT edges.tail_vertex FROM edges JOIN in_europe ON edges.head_vertex = in_europe.vertex_id WHERE edges.label = 'within' ), born_in_usa(vertex_id) AS ( SELECT edges.tail_vertex FROM edges JOIN in_usa ON edges.head_vertex = in_usa.vertex_id WHERE edges.label = 'born_in' ), lives_in_europe(vertex_id) AS ( SELECT edges.tail_vertex FROM edges JOIN in_europe ON edges.head_vertex = in_europe.vertex_id WHERE edges.label = 'lives_in' ) SELECT vertices.properties->>'name' FROM vertices JOIN born_in_usa ON vertices.vertex_id = born_in_usa.vertex_id JOIN lives_in_europe ON vertices.vertex_id = lives_in_europe.vertex_id;

4 lines in Cypher vs 29 in SQL. Different models for different use cases.

Triple-Stores & SPARQL

Equivalent to property graphs, different vocabulary. All data as three-part statements:

Subject
(a vertex)
Predicate
(edge or property)
Object
(value or vertex)

Example: (Jim, likes, bananas) — (Lucy, age, 33) — (Lucy, marriedTo, Alain)

@prefix : <urn:example:>. _:lucy a :Person; :name "Lucy"; :bornIn _:idaho. _:idaho a :Location; :name "Idaho"; :type "state"; :within _:usa. _:usa a :Location; :name "United States"; :type "country"; :within _:namerica.

SPARQL: Even More Concise

The same "find emigrants" query:

PREFIX : <urn:example:> SELECT ?personName WHERE { ?person :name ?personName. ?person :bornIn / :within* / :name "United States". ?person :livesIn / :within* / :name "Europe". }

Cypher borrowed its pattern matching from SPARQL.

SPARQL is a powerful tool even if the semantic web never happens.

Graph DBs ≠ CODASYL Redux

They look similar at first glance, but differ fundamentally:

CODASYL (1970s)

  • Schema restricts nesting types
  • Traverse access paths to find records
  • Children are ordered sets
  • All queries imperative

Graph DBs (today)

  • Any vertex → any vertex, no restrictions
  • Direct ID lookup or index search
  • No ordering requirement
  • Declarative languages (Cypher, SPARQL)

The Foundation: Datalog

Studied since the 1980s. Provides the foundation later query languages build upon.

Data as predicates:

name(namerica, 'North America'). type(namerica, continent). within(usa, namerica). within(idaho, usa). born_in(lucy, idaho).

Build complex queries from small reusable rules. Used by Datomic and Cascalog.

Datalog: Step-by-Step Rules

within_recursive(Location, Name) :- name(Location, Name). /* Rule 1 */ within_recursive(Location, Name) :- within(Location, Via), /* Rule 2 */ within_recursive(Via, Name). migrated(Name, BornIn, LivingIn) :- name(Person, Name), /* Rule 3 */ born_in(Person, BornLoc), within_recursive(BornLoc, BornIn), lives_in(Person, LivingLoc), within_recursive(LivingLoc, LivingIn). ?- migrated(Who, 'United States', 'Europe'). /* Who = 'Lucy'. */

Rules can reference other rules (like function calls). Small pieces compose into powerful queries.

Less convenient for one-off queries, but excels at complex data.

Summary

Chapter 2 Summary

Data started as one big tree (hierarchical model), then relational solved many-to-many. Now NoSQL diverges:

📄 Document

Self-contained data, rare relationships. Schema-on-read.

📊 Relational

Joins, normalization, many-to-many. Schema-on-write.

🔗 Graph

Anything related to everything. Flexible & evolvable.

Each is good in its domain. One model can emulate another — but the result is often awkward.
